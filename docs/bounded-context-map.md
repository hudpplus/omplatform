## 2a. 服务合并说明

> **2026-06-14 架构优化**：原 16 个微服务合并为 7 个部署单元。
> 合并原则：按调用亲密度 + 团队归属，保留独立变化边界。
> 详见：[服务合并决策 ADR](./adr/ADR-049-service-merging.md)

| 合并后服务 | 包含的原服务 | 合并理由 |
|-----------|-------------|---------|
| **oms-trade** | order-core, workflow-service, cart-service, order-query-service, notification-service | 下单链路五大组件协同度极高，workflow 编排的每一步都调用 order-core，cart 是下单前置，query 是同一聚合的查询端 |
| **oms-finance** | payment-core, aftersale-service | 资金域统一由资金团队维护；售后中的退款操作紧密依赖支付模块 |
| **oms-fulfillment** | inventory-service, logistics-service | 履约域由同一团队维护；发货操作同时涉及库存扣减和物流单创建 |
| **oms-marketing** | price-service, promotion-service, coupon-service, member-service | 计价管道 4 步骤紧密协作（basic→member→promotion→coupon），每次下单需全部参与，跨服务 RPC 开销不可接受 |
| **oms-channel-adapter** | channel-adapter | 防腐层独立，新增渠道变更频繁，需独立 CI/CD |
| **oms-risk-integration** | risk-integration | 防腐层独立，外部依赖隔离 |
| **内嵌模块** | id-generator → 库内嵌各服务 | Leaf/Snowflake 是算法库，做成独立服务让每次 ID 生成都走网络，不合理 |

## 3. 上下文映射关系

### 3.1 关系类型说明

| 类型 | 标记 | 含义 | 适用场景 |
|------|------|------|---------|
| **Customer/Supplier** | [C/S] | 下游（Customer）依赖上游（Supplier）的产出 | 数据依赖明确 |
| **Partnership** | [P] | 两个上下文紧密协作，互相依赖 | 联合变更频繁 |
| **Anti-Corruption Layer** | [ACL] | 建立防腐层防止外部模型污染 | 外部系统/渠道集成 |
| **Open-Host Service** | [OHS] | 定义共享协议供多方调用 | 编排层 |
| **Conformist** | [CF] | 下游无差异地遵循上游模型 | 通用域 |

### 3.2 映射矩阵

```
订单核心(BC-01)
  ├─→ 支付(BC-02)      [C/S] 依赖支付结果 → 推进订单状态机
  │                     共享: order_no, payment_no, PAID, REFUNDING
  ├─→ 库存(BC-03)      [P]   下单扣库存，取消/退款释放库存
  │                     共享: sku_id, quantity, hold_qty
  ├─→ 售后(BC-04)      [C/S] 售后引用订单状态，售后结果回写订单
  │                     共享: order_no, REFUNDING, REFUNDED, RETURNING
  ├─→ 营销价格(BC-05)  [C/S] 订单依赖计价/促销/优惠券结果
  │                     共享: amount, discount, coupon_no
  ├─→ 物流(BC-06)      [C/S] 订单依赖物流发货状态 → 推进状态机
  │                     共享: order_no, SHIPPED, DELIVERED, tracking_no
  ├─→ 购物车(BC-07)    [C/S] 购物车选品 → 订单创建
  │                     共享: sku_id, quantity, selected
  ├─→ 会员(BC-08)      [C/S] 会员信息/等级影响订单创建
  │                     共享: member_id, tier_level
  ├─→ 工作流(BC-09)    [OHS] 流程引擎定义编排协议，订单提供原子服务
  │                     共享: order_id, action, status
  ├─→ 风控(BC-11)      [C/S] 风控结果 → HOLD/CANCELLED
  │                     共享: order_no, HOLD, risk_level
  ├─→ 查询(BC-13)      [C/S] 订单核心提供数据源 → Canal 同步 ES
  └─→ 通知(BC-12)      [C/S] 订单发布事件 → 通知触发

支付(BC-02)
  ├─→ 订单核心(BC-01)  [C/S] 支付回调 → 订单状态推进
  ├─→ 售后(BC-04)      [C/S] 售后发起退款的执行
  ├─→ 三方支付(外部)   [ACL] 支付宝/微信差异封装
  └─→ 会员(BC-08)      [C/S] 支付完成 → 积分发放

库存(BC-03)
  ├─→ 订单核心(BC-01)  [P]   下单预占/扣减，取消释放
  ├─→ 售后(BC-04)      [C/S] 售后退款 → 库存回滚
  └─→ 渠道(外部)       [ACL] 渠道库存隔离 SPI

售后(BC-04)
  ├─→ 支付(BC-02)      [C/S] 退款执行
  ├─→ 库存(BC-03)      [C/S] 退款退货 → 库存回滚
  ├─→ 营销价格(BC-05)  [C/S] 退款 → 优惠券回退
  ├─→ 物流(BC-06)      [C/S] 退货物流/换货发货
  ├─→ 工作流(BC-09)    [OHS] Saga 退款/换货编排
  └─→ 通知(BC-12)      [C/S] 售后通知

全渠道接入(BC-10)
  ├─→ 订单核心(BC-01)  [ACL] 渠道订单 → 标准化 → 创建订单
  └─→ 外部渠道(天猫等) [ACL] 外部差异化适配

风控(BC-11)
  └─→ 外部风控平台     [ACL] 三级降级保护核心交易

工作流(BC-09)
  ├─→ 订单核心(BC-01)  [OHS] 编排订单创建/取消/确认收货
  ├─→ 支付(BC-02)      [OHS] 编排支付成功 Saga
  ├─→ 营销价格(BC-05)  [OHS] 编排计价步骤
  ├─→ 库存(BC-03)      [OHS] 编排库存操作
  └─→ 物流(BC-06)      [OHS] 编排发货步骤
```

### 3.3 核心映射模式详解

#### 模式 1：全渠道接入 → [ACL] → 订单核心

```
天猫 ──┐
京东 ──┤     ┌─────────────────┐     ┌──────────────┐
抖音 ──┤────→│  Channel Adapter │────→│  订单核心     │
拼多多 ─┤     │  (防腐层)        │     │  (标准化领域)  │
微信 ──┘     │  SPI + 6 适配器  │     │  order_core   │
POS  ──┘     └─────────────────┘     └──────────────┘

作用: 6 个外部渠道的订单模型 → 6 种映射规则
     渠道模型 (raw_order) → 中台标准订单模型
     OrderCreateRequest 是防腐层的输出契约
```

#### 模式 2：工作流 → [OHS] → 所有核心服务

```
                ┌─────────────────┐
                │  Workflow       │
                │  (流程引擎+YAML)│
                └──────┬──────────┘
          ┌────────────┼────────────┐
          │            │            │
   ┌──────┴─────┐ ┌───┴──────┐ ┌───┴──────┐
   │ 订单核心    │ │ 库存     │ │ 支付     │
   │ createOrder│ │reserve   │ │ charge   │
   │ cancelOrder│ │confirm   │ │ refund   │
   │ confirmPaid│ │release   │ │          │
   └────────────┘ └──────────┘ └──────────┘

作用: 工作流定义"做什么" (WHAT)
     原子服务实现"怎么做" (HOW)
     是标准的 OHS/Publish Language 模式
```

#### 模式 3：风控 → [ACL] → 订单核心

```
外部风控平台 ──┐
              ├──→ risk-integration ──→ 订单核心
本地黑白名单  ──┘    (三级降级: L0/L1/L2)

作用: 外部风控不可用时 → 降级为本地黑白名单判断
     风控平台差异 → 统一 check/review/decision 接口
```

## 4. 共享术语映射

以下术语在不同上下文中可能有不同的含义或数据结构。详见 [通用语言术语表](ubiquitous-language.md)。

| 术语 | 出现上下文 | 含义差异 |
|------|-----------|---------|
| `order` | 订单核心/支付/售后/物流 | 订单核心: 聚合根，含 items/payments；支付: 支付标的；售后: 售后服务关联对象 |
| `status` | 订单核心/支付/售后/库存 | 各上下文有独立的状态机，状态值域不同 |
| `refund` | 支付/售后/订单核心 | 支付: 资金原路返回；售后: 退款流程触发；订单: 状态推进到 REFUNDED |
| `amount` | 支付/价格/订单核心 | 支付: 实付金额；价格: 原始金额-优惠前；订单: 商品金额 |
| `sku_id` | 订单核心/库存/购物车/营销 | 同一 SKU ID，但各单位持有的库存语义不同（可用/预占/锁定） |

## 6. 上下文边界验证

### 6.1 按业务变化频率

| 上下文 | 变化频率 | 说明 |
|--------|---------|------|
| 订单核心 | 🟢 低 | 状态机稳定，极少新增状态 |
| 支付 | 🟡 中 | 新增支付渠道 → 迭代 |
| 营销价格 | 🔴 高 | 促销活动形态频繁变化 |
| 全渠道接入 | 🔴 高 | 新增渠道或渠道 API 变更 |
| 售后 | 🟡 中 | 业务迭代新增售后类型 |
| 风控 | 🟡 中 | 风控规则调整 |
| 会员 | 🟢 低 | 等级体系稳定 |
| 通知 | 🟡 中 | 新增通知渠道 |

边界合理性：变化频率差异大的上下文之间通过事件/接口隔离，达到独立演化目标。

### 6.2 按团队归属（Conway's Law）

```
oms-trade（订单核心/工作流/购物车/查询/通知）── 订单团队 (3-4人)
oms-finance（支付/售后）──── 资金团队 (2-3人)
oms-marketing（营销价格/会员）── 营销团队 (2-3人)
oms-fulfillment（库存/物流）─── 履约团队 (2人)
oms-channel-adapter + oms-risk-integration ── 中台基础团队 (2人)
```

**现状**: 2026-06-14 服务合并后，团队结构基本对齐部署边界。每个团队负责 1-2 个服务，跨团队沟通面从 256 条降为 49 条。
